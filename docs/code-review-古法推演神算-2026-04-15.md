## 4. 深度代码审查

### 4.1 严重问题 (Critical) 🛑

#### 1. 支付后端使用内存存储订单数据
- 📍 **位置**: `server/app.js:27`
- 📝 **描述**: 使用 `const orders = new Map()` 存储订单，服务重启后所有订单数据丢失。用户付款后如果服务重启，将无法查询到订单状态，导致用户付了钱但功能未解锁。
- 💻 **代码**:
  ```javascript
  // 简易订单存储（生产环境应使用数据库）
  const orders = new Map();
  ```
- 💡 **建议**: 至少使用 SQLite/JSON 文件持久化存储，生产环境应使用 MySQL/PostgreSQL。

#### 2. VIP 状态仅存储在 localStorage，极易绕过
- 📍 **位置**: `pay-success.html:165` 及 `script.js` 支付相关逻辑
- 📝 **描述**: 支付成功后仅设置 `localStorage.setItem("fortune_vip", "true")`。任何人只需在浏览器控制台执行 `localStorage.setItem("fortune_vip", "true")` 即可绕过付费，无需真正支付。
- 💡 **建议**: VIP 验证应该在后端进行，使用 token/签名机制。前端仅做展示，核心内容从后端获取。

#### 3. 支付宝密钥硬编码在配置文件中，且配置文件被提交到 Git
- 📍 **位置**: `server/config.js`
- 📝 **描述**: 虽然当前是占位符 `YOUR_APP_ID`，但配置结构鼓励直接在代码中填写密钥。一旦填入真实密钥并推送到 public 仓库，会造成严重安全泄露。
- 💡 **建议**: 使用环境变量（`process.env.ALIPAY_APP_ID`）或 `.env` 文件（加入 `.gitignore`）管理敏感配置。

#### 4. 支付回调通知地址使用 HTTP 且为本地地址
- 📍 **位置**: `server/config.js:35`
- 📝 **描述**: `notifyUrl: 'http://YOUR_DOMAIN:3000/api/pay/notify'` 使用 HTTP 而非 HTTPS，且生产部署时容易遗漏修改。支付宝要求通知地址必须为 HTTPS 公网地址。
- 💡 **建议**: 使用 HTTPS 地址，且通过环境变量配置。

### 4.2 重要问题 (Major) ⚠️

#### 1. PAY_API_BASE 硬编码为 localhost
- 📍 **位置**: `script.js` 支付模块 及 `pay-success.html:137`
- 📝 **描述**: `const PAY_API_BASE = "http://127.0.0.1:3000"` 硬编码在前端代码中。部署到 GitHub Pages 后，前端无法访问本地后端服务，支付功能完全不可用。
- 💡 **建议**: 根据环境动态配置，或使用相对路径 + 反向代理。部署说明中应明确标注。

#### 2. 紫微斗数排盘为简化伪算法
- 📍 **位置**: `script.js:81-100`
- 📝 **描述**: 紫微斗数排盘使用 `safeMod(seed + day * 3, 12)` 等简单数学运算模拟星曜排列，并非真正的紫微斗数算法。这意味着排盘结果不准确，但用户支付解锁的正是这个模块。
- 💡 **建议**: 如果收费，应实现真正的紫微斗数排盘算法，或在页面上明确标注为"AI 趣味解读"而非"紫微斗数"。

#### 3. 称骨算命农历转换不准确
- 📍 **位置**: `bazi-core.js:383-389`
- 📝 **描述**: 使用固定日期范围简单推算农历月份，没有考虑闰月和年份差异。中国农历每年节气日期不同，这种硬编码方式会导致部分用户结果偏差。
- 💡 **建议**: 使用专业的阳历转阴历库（如 `lunar-javascript`），或去掉称骨功能。

#### 4. CORS 配置过窄，部署后支付功能不可用
- 📍 **位置**: `server/app.js:9`
- 📝 **描述**: `cors({ origin: config.frontendUrl })` 限制为 `http://127.0.0.1:4173`。部署到 GitHub Pages 后（域名为 `yesmanv2.github.io`），CORS 会阻止所有请求。
- 💡 **建议**: 配置多个允许的 origin，或在部署时更新为实际域名。

#### 5. deploy.sh 中 git push -f 使用 HTTPS 硬编码
- 📍 **位置**: `deploy.sh:65`
- 📝 **描述**: `git push -f "https://github.com/yesmanv2/strongfortuneteller.git" gh-pages` 强制推送且仓库地址硬编码。如果未配置凭证缓存，每次都需要输入密码。
- 💡 **建议**: 使用 `git remote` 获取地址，或使用 SSH 方式推送。

#### 6. 粒子动画在移动端消耗性能
- 📍 **位置**: `index.html:436-517`
- 📝 **描述**: Canvas 粒子动画在低端手机上会造成卡顿。60 个粒子 + 两两连线的 O(n²) 算法在每帧执行，无节流。
- 💡 **建议**: 在移动端减少粒子数量或禁用动画。增加 `matchMedia('(prefers-reduced-motion)')` 检测。

### 6.2 最终结论

项目功能丰富，八字命理核心算法（`bazi-core.js`）实现扎实，UI 设计有赛博朋克特色。但存在以下关键问题需要关注：

1. **支付安全是最大隐患**：VIP 验证纯前端、订单存内存、密钥管理不当——如果要真正收费，这三个问题必须全部解决。
2. **紫微斗数算法为简化版**：作为付费核心功能，算法准确性不足可能引起用户投诉。
3. **部署架构有缺陷**：前端部署在 GitHub Pages，后端在本地，两者无法通信。支付功能在线上环境不可用。
4. **代码组织需改进**：`script.js` 过于庞大，应拆分模块化。

**建议优先级**：
1. 🛑 修复支付安全问题（如果计划收费）
2. ⚠️ 明确部署架构（前后端分离方案）
3. 💡 代码模块化拆分
